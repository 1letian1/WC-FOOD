# 食刻小馆项目分析

> 分析日期：2026-07-12
> 依据：根目录 `AGENTS.md`、《微信点餐小程序产品需求整理文档.md》及 `YuanXing/` 高保真原型。本文是开发前分析，不改变上述来源文件。

## 1. 项目定位与目标

食刻小馆是单店铺、用户与商家共用一个微信小程序的轻量点餐系统。第一版只打通“用户点餐—商家接单—制作—堂食取餐/商家自配送—完成”的闭环，优先保证业务正确、接口稳定、订单状态安全以及云服务器可部署。

明确不做：支付、独立骑手端、第三方配送、多门店、评价、会员营销、地图轨迹、消息队列、微服务、WebSocket 实时提醒和复杂权限系统。

## 2. 角色与权限

| 角色 | 登录方式 | 主要能力 | 数据边界 |
|---|---|---|---|
| 普通用户 USER | 微信 `wx.login` code 换取 openid | 浏览商品、购物车、地址、堂食/配送下单、查单、取消待接单订单、配送确认收货 | 只能操作自己的购物车、地址和订单 |
| 商家管理员 MERCHANT | 账号密码 + BCrypt | 工作台、接拒单、制作和履约、商品分类与商品管理、店铺状态设置 | 只能操作会话 `shopId` 所属数据 |
| 配送人员 | 第一版不设独立角色和账号 | 由商家管理员承担开始配送、标记送达 | 沿用 MERCHANT 权限 |

用户端和商家端是两个明确会话域，不允许靠前端隐藏按钮控制权限，也不自动通过微信身份识别商家。

## 3. 页面与功能理解

### 3.1 用户端

- 统一登录：协议确认、微信快捷登录、商家弱入口。
- 首页：营业状态、堂食/配送切换、公告、分类、商品列表、商品详情/规格口味选择、购物车入口。
- 购物车：组合项增删改查、金额预览、清空、结算；最终金额仍由下单接口重算。
- 堂食确认：联系人、电话、桌号或“暂未入座”、备注、商品清单。
- 配送确认：本人地址、收货人、电话、配送说明、起送金额和配送费。
- 订单：提交结果、列表、详情、状态进度、取消待接单订单、已送达确认收货、再来一单（后续便利功能）。
- 我的：用户信息、地址、订单、联系商家、协议和商家入口。

### 3.2 商家端

- 登录：账号密码、失败限制、记住账号仅由客户端保存账号、忘记密码提示。
- 工作台：今日订单/营业额及各待处理状态统计，营业/堂食/配送开关。
- 订单：按类型和状态筛选，按订单号、电话、用户名称搜索，执行独立状态动作。
- 商品：分类、商品新增编辑、逻辑删除、上下架、售罄、库存、规格、口味、图片。
- 店铺：名称、Logo、电话、地址、公告、营业时间、配送说明、起送金额、配送费及开关。
- 账号：修改密码、退出；修改密码后全部旧商家会话失效。

## 4. 核心模型

- 店铺与账号：`shop`、`merchant_account`。
- 用户与地址：`user`、`address`。
- 商品域：`category`、`product`、`product_specification`、`product_taste`。
- 交易前：`shopping_cart`；购物车价格仅作展示缓存，不是结算依据。
- 订单域：`orders`、`order_item`、`order_status_log`；订单明细保存名称、图片、规格、口味、价格快照。

所有金额使用 `DECIMAL(10,2)` / `BigDecimal`。MySQL 是正式业务数据源，Redis 仅承担会话、限流、序号、幂等与缓存。

## 5. 履约方式与完整状态机

稳定编码：`PENDING_ACCEPT=1`、`ACCEPTED=2`、`COOKING=3`、`READY_FOR_PICKUP=4`、`DELIVERING=5`、`DELIVERED=6`、`COMPLETED=7`、`CANCELLED=8`、`REJECTED=9`。

| 订单类型 | 合法主流程 | 异常流转 |
|---|---|---|
| 堂食 DINE_IN=1 | 1 → 2 → 3 → 4 → 7 | 1 → 8（用户取消）；1 → 9（商家拒单） |
| 配送 DELIVERY=2 | 1 → 2 → 3 → 5 → 6 → 7 | 1 → 8（用户取消）；1 → 9（商家拒单） |

配送 `DELIVERED → COMPLETED` 只允许订单所属用户确认；第一版商家不能跳过确认。状态更新必须使用预期状态和 `version` 条件更新，更新与状态日志、取消/拒单库存恢复同一事务，影响行数为 0 返回 409/60002。

## 6. YuanXing 目录结论

`YuanXing/` 是由 Figma 设计导出的、可通过 Vite 启动的 React 18 + TypeScript 高保真交互原型，不是微信小程序生产前端，也不是后端代码。目录包含：

- 单体 `src/app/App.tsx`，内置 21 个用户端/商家端屏幕、模拟数据、简单状态交互和 375×812 手机框。
- 约 49 个 TSX 文件，其中大部分是通用 Radix/shadcn UI 组件；6 个 CSS 主题文件。
- 两份原型生成提示/流程说明、一张 PNG 资源、Figma attribution 与 Vite 配置。

可复用程度：视觉 token、文案、页面信息层级、导航关系、状态标签和交互意图可高程度复用；React 页面代码只能作为前端实现参考。其数据为硬编码、路由和状态集中在单文件、没有 API/鉴权/持久化，不能直接作为微信小程序代码或后端业务依据。原型依赖较多，也不应反向决定后端依赖。

## 7. 冲突、遗漏与风险

### 7.1 已裁决的轻微冲突

| 项目 | 差异 | 保守裁决 |
|---|---|---|
| 产品名 | 原型为“食刻到家”，规范/需求为“食刻小馆” | 统一以“食刻小馆”为准 |
| 商家角色文字 | 需求称“店铺管理员”，规范只定义 MERCHANT | 第一版仅 `MERCHANT`，角色字段保留但不做复杂 RBAC |
| 商家完成配送单 | 产品角色概述含“完成订单” | 遵循严格状态机：商家只标记已送达，用户确认完成 |
| 复杂库存 | 需求称暂不做 | 不做仓储系统，但必须做单商品库存条件扣减和取消/拒单恢复 |
| 搜索/规格/改密 | 页面文档较完整，MVP 又列为可后续 | 数据结构与接口预留；核心闭环后按里程碑补齐 |
| 预计时间 | 原型展示预计用餐/送达时间 | 第一版可由店铺配置生成展示文案，不承诺精确调度 |
| 购物车切换履约模式 | 原型提示商品保留 | 购物车组合项保留，结算信息按当前模式重算 |
| 前端售价/小计 | 原型本地计算 | 仅展示；下单时后端查询有效商品、规格价格并重算 |

### 7.2 功能遗漏与实现风险

- 产品文档未完整定义规格是否加价；默认规格表包含 `price_delta`，下单价为商品基础价加规格差价，口味默认不加价。
- “月售”没有自然月统计口径；默认从已完成订单明细聚合，不把可篡改累计值作为交易依据，可后续缓存。
- 无支付却使用“实付金额”文案；字段保留为应付/订单金额，不增加支付状态和支付流水。
- 拒单原因、订单地址、商品信息必须存快照，否则历史记录会被后续编辑污染。
- 单店铺仍必须保留并校验 `shop_id`，为数据隔离和错误防护服务，不代表支持多门店。
- 默认地址唯一性无法仅靠普通唯一索引表达；在事务内清除同用户其他默认地址后再设置，并通过并发测试保障。
- 原型中存在演示密码和纯前端校验，只能视为演示数据，绝不能进入日志或生产初始化数据。
- 原型未覆盖所有异常/空状态的真实接口行为；以后端错误码和 HTTP 状态为准。

## 8. 安全与一致性基线

- 256 bit 安全随机 Token + Redis 固定 TTL；用户 7 天，商家 12 小时。
- 微信 code2Session 封装为外部 Client；openid 不接受前端直传；dev 模拟登录默认关闭、prod 禁止。
- 商家 BCrypt strength 12；15 分钟失败 5 次锁 30 分钟；密码和完整 Token 不记录。
- Jakarta Validation、参数化 SQL、排序白名单、上传类型/大小/MIME/路径校验。
- 下单使用客户端幂等键；Redis 原子占位结合订单唯一业务约束，不用分布式锁替代数据库约束。
- CORS 仅为本地 Swagger/管理调试开放明确来源；生产 API 通过 HTTPS/Nginx。
- 分类、商品、地址逻辑删除；订单、明细和状态日志永久保留。

## 9. 默认假设与待确认问题

### 9.1 不阻塞开发的默认假设

- 第一版固定一个初始化店铺，但所有核心表保留 `shop_id`。
- 联系电话按中国大陆 11 位手机号校验；若后续支持座机再放宽。
- 包装费第一版默认 0，可在订单中保留快照字段，不新增复杂计费规则。
- 商品规格单选、口味单选且可选；同商品+规格+口味构成购物车唯一组合。
- 库存按商品总库存管理，不拆到规格库存。
- “再来一单”只尝试把仍有效商品加入购物车，失效项提示并跳过。
- 联系商家由客户端调用店铺电话，不单独建立通信服务。

### 9.2 当前真正阻塞开发的问题

当前没有阻塞项目骨架、数据库基础结构、鉴权基础设施和核心订单闭环开发的问题。接入真实微信登录和生产部署前，必须由环境提供微信 AppID/Secret、正式域名/HTTPS 证书及服务器配置；这些属于部署凭据，不应写入仓库，也不阻塞本地 mock 开发。
